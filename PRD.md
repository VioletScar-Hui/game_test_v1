## 二、产品定位与目标

### 2.1 一句话定位

一款**美食主题、哥特荒诞幽默**的网页端 Survivors-like：玩家扮演 Spicy 家族成员，在被"永恒饥饿诅咒"笼罩的 Spiceland 生存 30 分钟，表面上追捕从未现身的反派 Gourmando Feastó。

### 2.2 设计支柱（决策时的取舍依据）

1. **数值滚雪球的爽感**：前 5 分钟求生 → 中期成型 → 后期清屏。任何新功能不能稀释这条曲线。
2. **构筑的可读性**：每局 6 武器 + 6 被动，进化配方明确可查（图鉴内可见），鼓励"这局凑某套"。
3. **荒诞叙事轻量化**：剧情只通过加载文案、物品描述、敌人台词传递，绝不打断战斗。深度剧情（第七章）全部走"可选收集"路线，不看不影响游玩。
4. **零安装即点即玩**：保持单 HTML 文件可分发，总资产体积控制在 50MB 内（见 8.2 压缩规范）。

### 2.3 目标体验指标

| 指标 | 目标 |
|------|------|
| 单局时长 | 30:00 强制结算（卫生检查员登场） |
| 首次死亡时间（新手） | 8-12 分钟（太快=挫败，太慢=无聊） |
| 一局升级次数 | 25-35 次（保证三选一频率） |
| 同屏敌人峰值 | ≥ 300（性能目标 60fps，见 9.5 对象池） |
| 完整解锁周期 | 15-20 局（不含真结局收集线，真结局 +10 局） |

## 三、核心玩法设计

### 3.1 局内循环（一局 30 分钟）

```
移动躲怪 → 自动攻击 → 杀敌掉经验糖 → 拾取升级
    ↑                                      ↓
  走位风筝 ← 强度提升 ← 三选一（武器/被动/金币）
                ↓
  满级武器 + 对应被动 + 开宝箱 → 进化武器 → 清屏
                ↓
  30:00 卫生检查员登场 → 击杀=Victory / 被杀=Game Over
                ↓
          结算金币 → 局外
```

### 3.2 局外循环

```
结算金币 → 永久强化商店（PowerUp） → 变强
       → 达成解锁条件 → 新角色 / 新关卡 / 新武器 / 新圣杯卡入池
       → 便条收集 → 拼出 Gourmando 真相 → 真结局挑战
       → 图鉴收集 → 重开一局
```

### 3.3 三幕节奏（对齐叙事圣经 1.2 节）

| 阶段 | 时间 | 玩法体验 | 叙事载体 |
|------|------|----------|----------|
| 第一幕：觉醒 | 0:00–5:00 | 弱怪、低密度、攒第一波构筑 | 环境叙事 ambientLore 滚动（第一阶段文案） |
| 第二幕：迷失 | 5:00–20:00 | 每分钟换波次、精英入场、宝箱进化、随机事件 | 精英死亡台词、便条掉落、第二阶段 ambientLore |
| 第三幕：清算 | 20:00–30:00 | 高密度怪潮、Boss 前奏事件 | bossHint → 卫生检查员登场台词、第三阶段 ambientLore |

## 四、功能需求

> 优先级定义：**P0** = 不做不算 Survivors-like；**P1** = 留存与重复可玩性；**P2** = 打磨与惊喜。
> F12–F17 为 v1.1 新增的深度功能，详细设计见第六、七章，此处只列需求边界。

### F1. 升级三选一系统（P0）

- 拾取经验糖升级时，**暂停游戏**，弹出 3 个随机选项卡（幸运值高可出第 4 个）。
- 选项池 = 未满级的已持有武器/被动升级 + 未持有的武器/被动（受槽位限制）+ 保底项（金币 25 / 鸡腿回血）。
- 权重：已持有项权重 ×2（VS 同款偏好）；同一面板不重复。
- 每个选项卡显示：图标、名称、等级 New/Lv.n→n+1、一行效果描述、一行荒诞 flavor 文案。
- UI 新增文件：`js/ui/render-levelup.js`；数据驱动自 `weapons-data.js` / 新增 `passives-data.js`。

### F2. 武器槽位与被动道具系统（P0）

- 玩家持有上限：**6 武器 + 6 被动**。HUD 左上角两行图标显示（含等级点）。
- 被动道具 8 件（数值见 5.3）：泡面（伤害）、健身护腕（攻速）、放大镜（范围）、外卖头盔（护甲）、空奶茶杯（磁铁）、跑鞋（移速）、幸运猫摆件（幸运）、祖传食谱（经验）。
- 改造 `js/entities/weapon-system.js` 支持多武器并行开火；新增 `js/entities/passive-inventory.js`。

### F3. 基础武器扩展至 8 把（P0）

### F4. 武器进化系统（P0）

- 条件：武器满级（Lv.8）+ 持有对应被动（任意等级）+ 击杀精英掉落的**神秘便当盒**。
- 开箱演出（F6）结束后武器图标替换为进化版，数值跃迁。
- 首发 4 条进化链（数值见 5.4）：

| 进化武器 | 配方 |
|----------|------|
| Divine Slipper 神圣拖鞋 | 弹跳拖鞋 Lv.8 + 跑鞋 |
| Excalibur Ladle 圣剑汤勺 | 旋转汤勺 Lv.8 + 泡面 |
| Meat Grinder 绞肉机 | 回旋菜刀 Lv.8 + 健身护腕 |
| Infinite Hot Pot 无限火锅 | 辣酱瓶 Lv.8 + 放大镜 |

- 图鉴内可查配方（已解锁显示全文，未解锁显示"？？？+ 剪影"）。

### F5. 分钟级波次表（P0）

- 用 30 行的波次时间轴替换现有"每种敌人独立 interval"刷法。每关一张表，结构：

```js
// js/config/waves-data.js
export const WAVES = {
  hungry_forest: [
    { minute: 0, enemies: [{ id: 'hangry_pigeon', interval: 800, count: 2 }], maxOnScreen: 30 },
    { minute: 1, enemies: [...], maxOnScreen: 50 },
    // ... 每分钟一行，minute 5/10/15/20/25 插入精英，minute 12 插入"怪潮事件"
    { minute: 29, event: 'boss', enemies: [] }
  ],
  // 其余 4 关同结构
};
```

- 特殊事件类型：`swarm`（一字横排冲屏的怪潮，VS 蝙蝠墙）、`ring`（圆环包围圈，VS 花圈）、`elite`、`boss`。
- 敌人血量随分钟缩放（公式见 5.5），波次表只管"刷什么、刷多密"。

### F6. 宝箱开箱演出（P1）

- 精英击杀必掉**神秘便当盒**（替代现直接奖励），拾取后全屏演出：便当盒打开 → 奖励图标滚动 → 定格。
- 奖励规则：有可进化武器 → 必出进化；否则随机升级 1 件（10% 概率 3 连，1% 概率 5 连）+ 金币。
- 新增 `js/ui/render-chest.js`；BGM 切换为开箱 jingle（见 8.8）。

### F7. 大地图与摄像机跟随（P1）

- 地图边界扩展为 **3840×2160**（3×3 屏），背景图平铺 + 边缘暗角；摄像机跟随玩家（`js/systems/camera.js` 已有 Camera 类，启用跟随模式即可）。
- 敌人在摄像机视野外 100–300px 环形带生成；超出 800px 的敌人传送回环形带（VS 同款防脱怪）。
- 背景资产改为**可平铺纹理**（见 8.6 资产规范第 2 类）。

### F8. 局外永久强化 PowerUp（P1）

- 主菜单新增"能力强化"入口，金币购买，跨局生效，可全额退款重配（VS 同款）。
- 8 项强化（价格曲线见 5.7）：最大生命、伤害、攻速、移速、磁铁、幸运、金币获取、复活次数。
- 存储于 localStorage 新键 `vs_powerups`，与存档槽独立。

### F9. 解锁链（P1）

| 解锁物 | 条件 | 叙事包装 |
|--------|------|----------|
| 角色 Imelda | 任意角色升到 Lv.10 | 默认仅 Antonio 可用，其余转为解锁制 |
| 角色 Gennaro | 累计击杀 3000 | 同上 |

## 六、游戏体验深度设计 ★

> 本章回答"玩 20 局之后凭什么继续玩"。原则：**深度来自决策点密度，不来自数值膨胀**。每个子系统都必须给玩家一个"这局我要不要换个玩法"的理由。

### 6.1 圣杯卡 Arcana 系统重做（F12，P1）

**现状问题**：圣杯卡作为局内商店商品出售，玩家用金币直接买强度，既无构筑决策也稀释金币经济。

**重做方案**（对齐 VS Arcana 体验）：

- **获取节奏**：开局选 1 张（从已解锁卡池随机抽 3 选 1）→ 第 11:00、21:00 天降"月亮糖"拾取后再各选 1 张。单局上限 3 张。
- **设计原则**：每张卡是**改变玩法规则的修饰器**，不是纯数值卡。卡面效果要让玩家"围绕这张卡改变本局构筑"。

### 6.4 难度层与无尽模式（F14，P2）

三层难度结构，全部是**通关后的自选项**，不影响新手曲线：

1. **辣度 1–5（Curse 滑条）**：选关界面的滑条，风险换收益（数值见 5.8）。HUD 显示辣椒图标数。
2. **加辣模式 Hyper**：每关独立解锁的开关，速度感与收益同步拉高，给熟练玩家"刷金效率档"。
3. **无尽模式 Limit Break**：30 分钟后的续局选项，满级武器可超限投点（数值见 5.8）。给构筑成型的玩家一个"看看这套能走多远"的舞台，也是排行榜数据来源。

### 6.7 升级面板操控（F18，P1）

VS 用 Reroll/Skip/Banish 把"被动接受随机"变成"主动管理随机"，是后期构筑深度的核心杠杆。照搬并绑定收集进度（次数表见 5.9）：

- **刷新**：整面板重抽（已放逐的选项不会回来）。
- **跳过**：放弃本次选择，补偿金币 10。
- **放逐**：点击选项卡上的 🚫 角标，该武器/被动本局不再出现——"我这局不要大蒜"成为可执行的决策。
- **封印**（局外）：图鉴里把不喜欢的武器/被动永久移出池（≤3 个，可随时解除），老玩家定向刷构筑的工具。

实现要点：四个操作全部收敛为 `upgrade-pool.js` 纯函数池构建的过滤参数，UI 只负责消耗次数 + 重新调用，无状态散布（见 9.3-F18）。

### 6.8 功能型武器（F19，P1）

现有 8 把武器全是输出型，后期构筑同质化风险高。引入"占槽但不打伤害"的功能武器（数值见 5.9），制造真实取舍：**第 6 个武器槽，要更多伤害还是要免死？**

- 外卖保温袋（护盾）/ 急冻冰箱门（冻结控场）/ 清桌服务铃（清屏保命）。
- 服务铃 Lv.1–5 会清掉怪物掉落（VS Pentagram 同款"代价设计"）——"铃铛到底值不值"会成为玩家社区的自发讨论题，这正是深度的表现。
- 三条进化链把功能武器推向后期 build 核心（米其林斗篷的免死、华丽月光珍珠的全图收割）。

### 6.10 挑战关与每日挑战（F21/F22，P2）

- **检查员大游行**（Boss Rush 型）：无小怪，每 90 秒连发一只精英/Boss 变体（共 10 波，含换色强化版），存活即胜。解锁条件：任意关 Hyper 模式通关。考验构筑成型速度而非挂机清屏。
- **鸽子王国**（怪海型）：全场只刷饥饿鸽子家族（换色烘焙出 5 个变种），密度 ×3、HP ×0.5——纯粹的爽局，也是连击系统（6.6-⑤）的最佳舞台。
- **每日挑战**：`seed = hash(UTC日期)`，当日全玩家同角色/同关卡/同辣度/同波次落点，本地记录最佳成绩。零服务器成本做出"全网同题竞争感"（结算页可截图分享，自带传播属性）。

### 6.11 物品稀有度评级（F25，P1）

Brotato 和 Death Must Die 都用稀有度把"每次升级"变成"每次开奖"——同一件武器/被动，掷中高稀有度时一步到位。本作美食化为 5 档食评（数值见 5.10）：

- 三选一的每张卡独立掷稀有度，卡面用边框颜色 + 食评标签呈现；高稀有度卡有金光描边 + 音效强调（复用 6.6 手感基建）。
- **让幸运值终于有存在感**：在此之前幸运只影响第 4 选项和掉落；引入稀有度后，Lucky Cat 被动、幸运 PowerUp、辣度收益全都指向"出货质量"，构成一条"堆幸运 → 赌传说卡"的玩法分支。
- 与进化解耦：传说档给的"半进化 30s"是临时预览，不替代正式进化（F4），避免破坏进化仪式感。
- 实现：`upgrade-pool.js` 抽出选项后追加 `rollRarity(luck, rng)` 纯函数；UI 读 `option.rarity` 决定边框与附赠。

### 6.12 Boss 机制化与弹幕预警（F26，P1）

**现状短板**：5.5 把 Boss 定义为"15000 血肉盾"，30 分钟高潮沦为站桩输出，是当前设计最大的体验洼地。对照 Death Must Die / Soulstone Survivors，Boss 应是"可读招式 + 走位考验"。

- **预警-招式两段式**：所有 Boss/精英招式先出预警（地面色块/轮廓闪烁 1–2s），再结算。玩家凭预警走位，把"DPS 检定"变成"DPS + 操作检定"。
- **卫生检查员三相**（数值见 5.10）：AOE 罚单圈 → 8 向"违规通知"弹幕 → 罚单激光扫屏（留安全缺口）。相位转换时全屏减速 0.3s（复用 timeScale）+ 台词气泡（复用 7.4 通道）。
- **精英轻量化**：每只精英 1 个标志招式即可（副厨僵尸"泼热油"扇形、图书管理员幽灵"甩书"直线弹），强化"剪影 + 招式"的记忆点。
- **可读性铁律**：预警色块用统一红橙色、与普通特效分层；弹幕密度受辣度缩放但永远留可走位空间（见 R12）。
- 实现：`config/boss-patterns-data.js` 声明式招式表（`{ telegraph, shape, params, damage }`）+ `systems/boss-director.js` 招式调度器；弹幕走对象池投射物，预警走粒子层。零新美术。

### 6.13 场景动态危害（F27，P1）

VS 的每张图都有专属环境威胁（炸花、闪电）。本作给每关一种主题危害，强化场景辨识度 + 逼迫走位：

| 关卡 | 危害 | 行为 |
|------|------|------|
| 饥饿森林 | 落食陷阱 | 随机区域天降整鸡，砸中范围伤害 + 短暂减速泥潭（可主动诱怪进去） |
| 华夫饼塔 | 糖浆瀑布 | 每 20s 一道竖直糖浆带横扫，碰到被黏住 1.5s |
| 食谱图书馆 | 飞书风暴 | 书架周期性喷出直线飞书弹幕（可被场景柱挡住） |
| 奶茶工厂 | 传送带 | 地面分区传送带持续推移玩家与敌人，改变走位惯性 |
| 神圣意面堂 | 吊灯坠落 | 预警后吊灯砸下，砸出持续灼烧的圣油地块 |

**附：厨具装备词缀掉落（F29，P2–P3，长线刷装层）**

借鉴 Halls of Torment 的装备 build：Boss/精英 ~5% 掉落可永久保留的厨具（围裙=防御 / 手套=攻击 / 厨师帽=功能 / 护身符=随机），4 槽局前装配，词缀随机（数值见 5.10）。定位明确为"给毕业玩家的可选刷装目标"，与核心 6+6 构筑解耦、可整体砍（见 R11）。装备存 `vs_gear`，掉落与词缀走种子随机（9.4-③）。

### 6.14 局外星级天赋树（F28，P2）

线性 PowerUp（F8）是"无脑变强"，缺少**取舍**。对照 Soulstone Survivors 天赋树 / Halls of Torment traits，新增一棵分叉树，用独立货币「星级评价★」（金币管 PowerUp，星级管天赋，两者分离），制造"这周目走攻击线还是财富线"的长线构筑（数值见 5.10）：

- 三主干：**炎之灶台**（攻：伤害/暴击/灼烧）、**铁之餐盘**（守：HP/护甲/免死）、**金之收银台**（财：金币/幸运/稀有度权重）。
- 2 个跨线 keystone 需点亮两条主干各 3 节点才解锁，是 build 身份的来源（如「光盘行动」「饥饿营销」，效果见 5.10）。
- 与 PowerUp 互补不互斥：PowerUp 给地板（每局都有的基础属性），天赋树给天花板与流派（keystone 改变玩法）。
- 实现：`config/skilltree-data.js`（节点 = `{ id, branch, cost, requires:[], effect }`）+ `ui/render-skilltree.js`；effect 注入 9.3-F2 的 `recalcStats` 同一乘区，不新开计算路径。

## 七、剧情深度设计 ★

> 原则（设计支柱 3）：深度剧情全部走**可选收集**路线。不看剧情的玩家体验零损失；想挖的玩家能挖出一条完整暗线和一个真结局。所有文本集中于 `js/config/narrative-data.js`（见 9.2）。

### 7.1 三层叙事结构

| 层 | 载体 | 受众 | 状态 |
|----|------|------|------|
| 表层：荒诞喜剧 | 加载文案、ambientLore、敌人死亡台词 | 100% 玩家 | ✅ 已有，本章做"分阶段化"增强 |

### 7.2 便条收集系统（F16，P2）

**核心笑点升级**：表层是"每关都找不到 Gourmando"；便条线揭示**他在故意留下线索引导 Spicy 家族——因为他需要有人找他**。

掉落规则：每关精英首杀必掉 1 张固定便条 + Boss 击杀掉 1 张；诅咒祭坛/账单堆小概率掉随机张。已获得的便条不重复（掉金币替代）。图鉴新增"便条"页，按编号排列，未集齐显示烧焦残页。

**12 张便条全文**（编号即收集顺序提示，文风对齐叙事圣经）：

### 8.2 通用图像规范（所有 PNG 适用）

| 项 | 规范 | 原因 |
|----|------|------|
| 格式 | PNG-24 | 现有管线统一 |
| 背景 | **纯白 #FFFFFF**（不要透明、不要浅灰、不要渐变白） | 运行时 `processSpriteImage` 从四边 flood-fill 去白，阈值 RGB≥230 |
| 主体禁忌 | 主体内部**不得有连接到边缘的大面积纯白/近白色块**（如白衣服、白高光糊边） | 会被抠图算法误删；近白请压到 RGB≤220 |
| 主体描边 | 建议 2-4px 深色描边 | 抗去白羽化，游戏内轮廓清晰 |
| 体积 | 单图 ≤ 500KB（背景 ≤ 1.5MB）；交付前过 `pngquant --quality 65-85` 或 tinypng | 当前背景单张 2.6-3MB，5 张背景已占 14MB，必须压缩 |
| 美术风格关键词（全项目统一） | 厚涂卡通（painterly cartoon）、哥特暗色环境 + 高饱和食物色、轻微赛璐璐描边、无文字水印 | 与现有资产一致 |

### 9.1 技术栈总览

| 层 | 选型 | 理由 / 约束 |

| 语言 | 原生 JavaScript（ES2022 Modules） | 零依赖、零构建即可开发；**不引入 TypeScript/框架**（现有 60+ 文件迁移成本 > 收益） |


### 9.4 关键技术专题

**① 全局时间缩放（hit-stop 与慢动作的统一基建）**
`game-core.js` 增加 `this.timeScale = 1.0`；`update(dt)` 入口处 `dt *= this.timeScale`。hit-stop 即 `timeScale=0` 持续 N ms（用真实时间恢复）；Boss 登场慢动作 = `timeScale=0.3` 渐变回 1。UI 与音乐不受 timeScale 影响（在缩放前分发）。

**② 精灵变体预烘焙（白闪 / 换色敌人 / 镜像 Boss 共用一套）**
加载完成后对需要变体的精灵离屏生成：白色版（受击闪白）、色相偏移版（隐藏关换色怪）、负片版（镜像 Boss）。运行时只 `drawImage` 缓存 canvas，**禁止逐帧实时 `ctx.filter`**（性能黑洞）。统一入口放 `assets.js` 新函数 `bakeVariant(sprite, type)`。

**③ 确定性随机（可复现的一局）**
新增 `js/systems/rng.js`：mulberry32 种子随机器。开局生成 `runSeed` 并入存档；波次落点、三选一抽取、掉落判定全部用 `rng()` 替代 `Math.random()`。收益：bug 可复现、读档不刷选项、远期可做"每日挑战"（全员同种子）。

**④ 碰撞与寻敌（300 敌人的成本控制）**
空间网格哈希（cell 128px）：每帧重建 `Map<cellKey, Enemy[]>`（重建比增量维护简单且 300 实体足够快）。武器寻敌（最近敌）、投射物碰撞、玩家受击全部走网格 3×3 邻域查询，把 O(n²) 压到 O(n)。落地文件：`systems/spatial-grid.js`，M1 期间随 F5 一起做（见 R3）。

**⑤ 文本渲染缓存**
中文 `fillText` 开销高。伤害数字（高频）用预渲染数字图集；台词/便条等低频文本首次渲染进离屏 canvas 缓存，淡出动画只变 `globalAlpha`。

**⑥ 输入序列检测（秘籍系统基建）**
`InputManager` 增加 32 长度环形按键缓冲；秘籍 = 声明式按键串列表（存 `config/`），每帧做尾部匹配。《禁忌菜单》遗物解锁后开放可视化输入界面；秘籍效果（解锁皮肤/曲目等）走事件总线，与正常解锁同一通道，不开后门。

**⑦ 日期种子与可分享对局**
每日挑战种子 = `fnv1a('2026-06-10')`，与 ③ 的 mulberry32 衔接；同种子保证当日全玩家波次落点/三选一/掉落完全一致。附带收益：任意一局的 `runSeed` 显示在结算页，玩家可手动输入种子复现或分享开局。

**⑧ 预警-结算两段式招式系统（Boss 招式 / 场景危害共用）**
Boss 招式（6.12）与场景危害（6.13）共享一套调度：每个招式声明 `{ telegraph: {shape, duration, color}, resolve: {shape, damage, params} }`。`boss-director` / hazard 在 telegraph 阶段只画预警层（粒子/半透明几何），duration 结束转 resolve 生成对象池投射物或 AOE 判定。统一两段式保证"所有威胁都先预警"的可读性铁律，也让两套系统复用同一渲染与判定代码。预警层颜色统一红橙并独立分层，与普通命中特效区分（见 R12）。

**⑨ 稀有度与词缀的种子化掷点**
稀有度（F25）与装备词缀（F29）的掷点全部经 ③ 的种子随机器，与三选一抽取同源。收益：每日挑战里"谁出了传说卡"对所有玩家一致；装备掉落可复现，便于平衡测试与 bug 复盘。`rollRarity` / `rollAffixes` 均为纯函数（入参 luck + rng），进 `node:test`。

### 9.5 性能预算（同屏 300 敌 + 200 投射物 @60fps）

- 敌人/投射物/掉落物/伤害数字全部上**对象池**（`entity-manager.js` 扩展 acquire/release，禁止战斗中 new）。
- 碰撞检测走 9.4-④ 空间网格。
- 经验糖 >300 自动合并（见 5.1）；视野外敌人跳过渲染（仍更新位置）。
- 每帧渲染排序只做一次（按 y 排序伪深度），用插入排序（帧间近有序）。
- 抠图管线 `processSpriteImage` 仅在加载时执行一次并缓存 canvas——现状已满足，新资产沿用即可。
- 性能护栏：开发模式 HUD 显示 fps + 实体数（fps 显示已有）；CI 冒烟脚本跑 1000 帧模拟更新（无渲染）断言耗时上限。

### 9.7 构建与分发管线

- 开发态：`index.html` + ES Modules 直接跑（需本地静态服务器，`python -m http.server` 或 VSCode Live Server）。
- 分发态：`build.ps1` 拼接产出 `bundle.js` → 内联 `vampire_survivors.html`。**v1.1 起 build.ps1 需扩展**：① 新 config 文件加入拼接清单（按 9.2 目录全量 glob，不再手工维护列表）；② 可选 `-Inline Images` 开关把 `assest/` 图片转 base64 内联（体积 ×1.33，配合 8.2 压缩红线评估是否启用）；③ 产物冒烟：构建后用 headless 方式（或手动）确认无 404 与控制台报错。
- 远期（不在本 PRD 范围）：若上线 Web 托管，改为 zip 整包 + 相对路径，单文件版仅作离线分发。

### 9.8 开发者工具与秘籍系统

第五章的数值调优需要快速验证手段，M1 起内置开发面板（VS 的 secrets 菜单本身就是面向玩家的彩蛋，开发工具与玩家秘籍共用一套基建）：

- **入口**：URL 参数 `?dev=1`，或秘籍输入（9.4-⑥）。
- **能力**：时间跳转（+5 分钟）、直给武器/被动/卡并指定等级、敌人密度滑条、无敌开关、显示碰撞盒/空间网格、显示当前 `runSeed`。
- **实现**：`js/systems/dev-panel.js`，所有操作通过事件总线发指令、不直接改游戏状态——作弊路径与正常路径同构，作弊能跑通 = 正常逻辑大概率没坏。
- **玩家侧秘籍**（《禁忌菜单》遗物后开放）：彩蛋码仅解锁换色皮肤、点歌机曲目等纯装饰内容；**不开放数值类秘籍**，保护成就与每日挑战的可信度。
